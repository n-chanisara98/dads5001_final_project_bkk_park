## Issues and Motivation 

เกิดจากการที่เราสังเกตเห็นเทรนด์ว่าคนเมืองในกรุงเทพฯ หันมาใส่ใจสุขภาพกันมากขึ้นอย่างเห็นได้ชัด ถ้าเราไปสวนสาธารณะหลักๆ เช่น สวนลุมพินี สวนจตุจักร ในช่วงเย็น เราจะเห็นภาพคนไปวิ่ง ไปเต้นแอโรบิคกันอย่างหนาแน่น เราจึงเลยมองว่า จริงๆ แล้วในกรุงเทพฯ ยังมีพื้นที่สีเขียวอีกหลายที่ ที่กระจายอยู่ตามจุดต่างๆ ซึ่งคนอาจจะยังไม่รู้ หรือยังเข้าไม่ถึง 

จากการวิเคราะห์ พวกเราพบว่าปัจจัยหลักๆ ที่คนเมืองจะเลือกไปใช้สวนสักที่หนึ่ง ประกอบด้วย 3 เรื่องสำคัญ
1.	สิ่งอำนวยความสะดวก เช่น สวนมีห้องน้ำไหม มีที่จอดรถไหม
2.	การเข้าถึง หรือความสะดวกในการเชื่อมต่อรถไฟฟ้า 
3.	ปัจจัยเรื่องฝุ่น PM2.5 

## Objective

จากแนวคิดทั้งหมดนี้ ทางกลุ่มจึงสร้างแพลตฟอร์ม Interactive Park Finder ขึ้นมา เพื่อเป็นเครื่องมือให้ผู้ใช้งานสามารถ 'เลือกและคัดกรองสวน' ตามเงื่อนไขความสะดวกของตัวเอง ควบคู่ไปกับการเช็กค่าฝุ่น PM2.5 ล่าสุด ณ สวนแห่งนั้น เพื่อช่วยในการตัดสินใจ และส่งเสริมให้คนกรุง ออกไปใช้พื้นที่สีเขียวได้อย่างมั่นใจและปลอดภัยที่สุด

## Solution (Methodology)

Database Architecture

จัดเก็บข้อมูลแยกเป็น 2 ส่วน

o	ส่วนแรก คือ ข้อมูลที่มีโครงสร้างชัดเจนอย่าง ตารางข้อมูลของสวนสาธารณะ รายละเอียดสถานีรถไฟฟ้าทั้งหมด จะเก็บและดึงมาจาก Snowflake 

o	ส่วนที่สอง คือข้อมูลค่าฝุ่น PM2.5 จากสถานีตรวจวัดในเขตต่างๆ ซึ่งมีลักษณะเป็น Dynamic JSON จาก external source จะเก็บใน MongoDB ซึ่ง Logic ในการทำงานของระบบคือ ทันทีที่มีผู้ใช้งานเปิดหน้า streamlit ขึ้นมา หลังบ้านจะวิ่งตรงไปยิง API ของทาง Air4Thai เพื่อดึงค่าฝุ่น ณ วินาทีนั้น จากนั้นเราจะนำ JSON นั้นบันทึกลงไปใน MongoDB แบบเรียลไทม์ โดยใช้วิธี replace_one หรือเรียกง่ายๆ ว่าเอาข้อมูลใหม่ล่าสุด 90 สถานี ไปอัปเดตทับข้อมูลเดิมเรื่อยๆ ดังนั้น เราจะได้ข้อมูลที่เป็นปัจจุบันที่สุดเสมอ

Processing 

o	ในแง่ของการประมาณผล เราใช้ DuckDB เข้ามาช่วย ในฐานะ In-Memory SQL Query Engine ทำหน้าที่เชื่อมโยงและคัดกรองข้อมูลจากทั้งสองฐานข้อมูลเข้าด้วยกันทันทีบนหน้าเว็บ ทำให้ทุกครั้งที่เรากดตัวเลือก หรือ filter Pandas DataFrame จะไม่ต้องต้องคอยสั่งกรองข้อมูลใหม่ 

o	สำหรับค่าฝุ่น ระบบจะทำการดึงข้อมูลพิกัดละติจูด-ลองจิจูดของสวนจาก Snowflake มาคำนวณเปรียบเทียบกับพิกัดสถานีของ Air4Thai ที่เก็บอยู่ใน MongoDB โดยใช้ฟังก์ชัน cdist จากไลบรารี SciPy เพื่อจับคู่ระยะทาง หาว่าสวนสาธารณะนั้นๆ อยู่ใกล้กับสถานีวัดฝุ่นสถานีไหนมากที่สุด เพื่อให้ได้ค่าฝุ่นที่สะท้อนความเป็นจริงของสวนนั้นมากที่สุด

AI Integration

o	Core Engine หลัก เลือกใช้โมเดล Gemini 2.5 Flash API 

o	AI ทำงานร่วมกับฐานข้อมูลจริง ผ่านกระบวนการที่เรียกว่า Context-Aware Prompt Engineering ร่วมกับการให้ระบบหลังบ้านจะวิ่งไปดึงข้อมูลจริงของสวนจาก Snowflake นำมาเข้า Rule-based Logic ก่อน ก่อนจะใส่เข้าไปในตัว Prompt เพื่อส่งให้ Gemini เพื่อ control ให้คำตอบอิง database จริง และคำตอบออกมาเกี่ยวข้องและเหมาะสมกับหัวข้อการออกกำลังกาย และสุขภาพ
